# Gatekeeper AI Roadmap Phase 5

## 1. Vision

Gatekeeper AI should evolve from a project management platform into an AI Business Operating System.

The current product already combines:

- project workspaces
- scoped conversations
- document retrieval
- project memories
- task tracking
- dashboard aggregation
- rule-based project intelligence

Phase 5 is the transition from a useful project-centric AI application into an operating layer that helps run real business workflows.

The core shift is:

- from storing project activity
- to structuring project meaning
- to coordinating project decisions
- to orchestrating business operations through shared intelligence

In this future state, Gatekeeper AI does not simply help a user find information. It helps the business:

- remember context
- detect risk
- track momentum
- understand relationships
- route work to the right specialized agent
- automate recurring operational follow-up

The system remains project-first. Every future layer should continue to orbit the project as the primary operational unit.

## 2. Milestone 1

## Project DNA

### Purpose

Project DNA is the durable identity layer for every project.

It should define what a project fundamentally is, beyond its current documents, conversations, and tasks.

Project DNA turns a project from a container of activity into a stable operational profile.

### Core Attributes

Future Project DNA should describe attributes such as:

- project type
- project objective
- current stage
- stakeholders
- decision style
- operating constraints
- preferred communication tone
- commercial posture
- legal sensitivity
- marketing posture
- expected timeline pattern
- strategic importance

These attributes should remain durable across multiple sessions and should not be treated as temporary chat artifacts.

### Relationship with Project Intelligence

Project Intelligence explains the project’s current state.

Project DNA should explain the project’s enduring identity.

Relationship model:

- Project Intelligence asks: What is happening now?
- Project DNA asks: What kind of project is this?

Project Intelligence should eventually read Project DNA as a baseline and then adjust scores relative to that baseline.

Examples:

- a high-risk legal matter and a normal marketing campaign should not be scored with identical expectations
- a project with known fast-turn constraints should interpret open tasks differently than a long-horizon asset project

### Future Uses

Project DNA can later power:

- smarter project intelligence thresholds
- better dashboard grouping
- role-specific agent behavior
- workflow automation triggers
- more accurate recommendations
- project-type-specific templates and playbooks

## 3. Milestone 2

## Project Timeline

Project Timeline is the chronological memory layer that turns scattered activity into a readable operational history.

### Sessions

Every project-scoped conversation should contribute to a session timeline.

The timeline should represent:

- when work happened
- what was discussed
- what changed in that session
- what was promoted into durable memory

### Tasks

Tasks should appear in timeline order when they are:

- created
- updated
- completed
- reopened

This makes the timeline reflect execution, not just discussion.

### Documents

Documents should appear when they are:

- uploaded
- linked to the project
- indexed successfully
- replaced or removed

That creates a visible chain of evidence for project knowledge growth.

### Decisions

Decision memories should become first-class timeline events.

They matter because decisions are often more strategically important than ordinary conversations or task creation.

### Marketing

Marketing-related outputs should eventually be included in the project timeline when they are project-linked.

This may include:

- content packages
- campaign drafts
- listing narratives
- publishing milestones

### Chronological History

The timeline should become the operational story of the project.

It should answer:

- what happened first
- what changed next
- what decisions were made
- what tasks followed
- where the project stalled or accelerated

The Project Timeline is a bridge between raw memory and explainable business context.

## 4. Milestone 3

## Knowledge Graph

The Knowledge Graph is the relationship layer that connects isolated project records into a business-wide network of meaning.

### Relationship Between

#### Projects

Projects should be linkable to:

- other related projects
- parent or child workstreams
- recurring campaigns
- shared stakeholders

#### Owners

Owners should become reusable entities linked to:

- projects
- decisions
- tasks
- communications

#### Buyers

Buyers should connect to:

- project opportunities
- negotiation history
- follow-up tasks
- legal or financial context

#### Documents

Documents should be connected not only to projects, but eventually to:

- decisions they support
- tasks they triggered
- legal entities they mention
- marketing deliverables derived from them

#### Tasks

Tasks should connect to:

- decisions
- sessions
- project stages
- responsible parties

#### Marketing

Marketing objects should be linkable to:

- projects
- property narratives
- target audiences
- content outputs
- campaign outcomes

#### Legal

Legal objects should connect to:

- agreements
- disputes
- approvals
- constraints
- risk signals

#### Properties

Properties should eventually become graph nodes connected to:

- projects
- owners
- buyers
- documents
- legal matters
- marketing outputs

### Future Graph Queries

The long-term value of the graph is not storage, but query power.

Examples of future graph questions:

- Which projects involve the same owner?
- What decisions are repeatedly associated with stalled projects?
- Which documents influenced the latest legal or commercial decisions?
- What tasks are linked to a specific property across multiple projects?
- Which marketing outputs came from the same project evidence base?

The Knowledge Graph should be built only after the current project-first intelligence layers are stable enough to avoid premature complexity.

## 5. Milestone 4

## Specialist AI Agents

Specialist AI Agents are role-oriented intelligence workers that operate on shared project context instead of isolated prompts.

### Marketing Agent

Purpose:

- generate campaigns
- refine messaging
- propose content plans
- reuse project facts without hallucinating unsupported claims

### Legal Agent

Purpose:

- surface legal issues
- organize document-backed concerns
- highlight unresolved approvals or constraints
- support structured legal follow-up

### Property Agent

Purpose:

- organize listing intelligence
- summarize project property context
- surface buyer/seller positioning
- support property-specific operations

### Investment Agent

Purpose:

- frame opportunities
- support investor-facing summaries
- highlight commercial signals, risks, and next actions

### Drone Survey Agent

Purpose:

- interpret survey-related project material
- connect visual or field evidence to project workflow
- support documentation and follow-up workflows around site intelligence

### Shared Intelligence Model

Every agent should read the same Project Intelligence rather than recomputing its own independent worldview.

That means agents should consume shared context such as:

- project health
- project confidence
- project priority
- project risk
- project momentum
- recommended next action
- Project DNA when it exists
- timeline context when it exists

This avoids duplicate logic and keeps agent behavior aligned across the system.

The rule is:

- one underlying intelligence model
- many role-specific readers and actors

## 6. Milestone 5

## Automation

Automation is the operational execution layer that turns intelligence into repeatable business movement.

### Notifications

Future automation should notify users about:

- overdue tasks
- stalled projects
- new high-risk signals
- missing follow-up after important decisions

### Daily Briefing

Gatekeeper AI should eventually generate a daily operational briefing based on:

- dashboard signals
- updated project intelligence
- tasks due today
- new decisions learned
- fresh session summaries

### Calendar

Calendar alignment should eventually connect project work with:

- due dates
- reminders
- milestones
- recurring operational routines

### Publishing

Content Studio and future marketing workflows should support automation such as:

- queueing content
- preparing review-ready assets
- planning publishing cadence

### Scheduled Reports

The system should eventually provide scheduled reports for:

- project health changes
- priority changes
- new decisions
- new summaries
- business-level operational snapshots

### Follow-up Reminders

The platform should eventually remind users to act when:

- a project has unresolved open tasks
- recent activity has stopped unexpectedly
- document context is insufficient for current work
- project confidence is too low for reliable execution

Automation should be introduced only after the rule-based intelligence layer is stable enough to avoid noisy or low-trust actions.

## 7. Long-term Vision

The long-term architecture progression is:

Mission Control

↓

Project Intelligence

↓

Project DNA

↓

Knowledge Graph

↓

AI Agents

↓

Business Operating System

This is not a sequence of unrelated features. It is a stack.

Each layer depends on the one below it:

- Mission Control surfaces what matters now
- Project Intelligence interprets project state
- Project DNA stabilizes project identity
- Knowledge Graph connects entities and relationships
- AI Agents act through shared intelligence
- the Business Operating System emerges when all of these layers coordinate work coherently

---

## Engineering Principles for Phase 5

Phase 5 planning should continue to follow these constraints:

- keep the project-first architecture
- reuse existing intelligence
- avoid duplicate logic
- prefer rule-based systems before AI reasoning
- keep one responsibility per module
- avoid schema changes in roadmap planning
- treat this document as planning only, not implementation

Phase 5 should extend the current foundation rather than bypass it.

That means:

- project memory should remain central
- project intelligence should remain explainable
- dashboard aggregation should remain composable
- future agents should share intelligence instead of inventing parallel logic

## Closing Direction

Gatekeeper AI should evolve deliberately.

The product should not jump directly into autonomous agents or broad automation without first strengthening:

- structured project identity
- chronological business memory
- explicit relationships
- trusted scoring
- shared operational context

Phase 5 is the roadmap for that evolution.

It is the bridge from a strong AI project workspace into an AI-native Business Operating System.
